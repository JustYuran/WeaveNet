### 📜 Логика создания файла списка изменений (ChangeLog)

При каждом завершении этапа разработки (после выполнения пункта из `TODO.md`) создавай новый файл в папке `/workspace/ChangeLog/`, строго следуя алгоритму ниже:

####  1. Версионирование (5-частный формат)
- **Формат:** `vA.B.C.D.E` (например, `v0.0.0.0.1`).
- **Инкремент:** Увеличивай последнюю цифру на `1`.
- **Правило переноса (каскадное):** Если последняя цифра достигает `9`, она сбрасывается в `0`, а цифра слева от неё увеличивается на `1`. 
  - Пример: `0.0.0.0.9` → `0.0.0.1.0`
  - Пример: `0.0.0.1.9` → `0.0.0.2.0`
  - Пример: `0.0.1.9.9` → `0.0.2.0.0`
- **Имя файла:** `WeaveNet_ChangeLog_v{версия}.txt`

#### 📝 2. Заполнение разделов (по шаблону)
| Раздел | Что писать |
|--------|------------|
| `Дата:` | Текущая дата и время создания (формат: `ДД.ММ.ГГГГ ЧЧ:ММ`). |
| `Выполненные задачи:` | Краткий список пунктов из `plan.txt`/`TODO.md`, реализованных в этом обновлении. Используй маркированный список. |
| `Технические детали:` | Суть изменений в архитектуре, алгоритмах или оптимизациях. Без кусков кода, только логика и геймплейные последствия. |
| `Статус сборки:` | Оценка готовности: `✅ Стабильная`, `⚠️ Требует тестирования`, `🔧 Экспериментальная`. |
| `Затронутые файлы:` | Список относительных путей к файлам, которые были созданы или изменены (например: `/src/map/HexGrid.ts`, `/workspace/Rules/plan.txt`). |
| `Предыдущие версии:` | **Блок поддержки истории.** Вноси сюда краткую выжимку из **последних 5 файлов** ChangeLog. Формат записи: `v0.0.0.0.X: [1-2 предложения о сути]`. При создании новой версии добавляй текущую в начало списка и удаляй самую старую (6-ю), чтобы блок всегда содержал ровно 5 записей. |

#### 🔄 3. Порядок действий при генерации
1. Просканируй папку `/workspace/ChangeLog/`, найди файл с максимальной версией.
2. Создай новый файл с инкрементированной версией.
3. Заполни разделы `Выполненные задачи`, `Технические детали`, `Статус сборки`, `Затронутые файлы` на основе проделанной работы.
4. Сформируй блок `Предыдущие версии`: возьми заголовки и краткие описания из 4 последних файлов + добавь описание текущей версии.
5. Сохрани файл. Убедись, что в папке нет дублей или пропущенных версий.

Эта логика гарантирует прозрачность истории разработки, упрощает откат к предыдущим состояниям и помогает быстро оценивать прогресс без чтения всего кода.
