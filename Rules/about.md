# 📘 WeaveNet: Дизайн-документ и Архитектура Проекта

## 🌍 Концепция
`WeaveNet` — браузерная стратегическая песочница, где игрок разворачивает децентрализованную mesh-сеть в условиях сложного рельефа. Игра не имеет жёстких условий победы; прогресс определяется масштабом, устойчивостью и оптимизацией сети. Проект разрабатывается модульно, с чётким разделением игровой логики, рендеринга и системных менеджеров.

---

##  Игровые Механики (Design)

### 1. 🗺️ Мир и Карта
- **Гексагональная сетка:** Все координаты, расстояния и радиусы измеряются в гексах. Один гекс = одна клетка поля.
- **Форма:** Карта визуально повторяет территорию России.
- **Биомы:** Равнина (база), Пустыня (помехи/перегрев), Снег (риск разряда/поломок). Генерируются кластерами.
- **Преграды:** Горы (блок сигнала/стройки), Пропасти (разрыв связи), Вода (блок стройки). Занимают целые гексы.

### 2. 🏗️ Инфраструктура
- **Габариты:** Все узлы занимают ровно 1 гекс.
- **Типы:** Роутер (треугольник, средний радиус), Вышка (пятиугольник, большой радиус), Генератор (производит энергию).
- **Режимы:** ⚪ Белый (выкл), 🟡 Желтый (50% радиус, экономия), 🟢 Зеленый (100% радиус, стандарт), 🔴 Красный (блок/нет энергии).
- **Визуал:** Покрытие — полупрозрачный градиентный круг. Зоны суммируются при пересечении.

### 3. 👥 Население
- **Размещение:** 0–6 пользователей на гекс. Не спавнятся на преградах или занятых клетках. 1.5*(число гексов на карте)=число пользователей в мире.
- **Статусы:** 
  - ⚪ Белый (нейтрален), 🟡 Желтый (заражен, радиус 2 гекса, +0.5 инфы/с, высокий риск поломки), 🟢 Зеленый (Подключен, радиус 2 гекса, +1.0 инфы/с, низкий риск), 🔴 Красный (блок/обрыв).
- **Циклы:** Жёлтый → Красный → Белый (вирус теряется). Зелёный → Красный → Зелёный (сохраняет лояльность, 1-3% шанс сброса до белого).

### 4. 💰 Экономика
- **Информация:** Валюта строительства. Доход = Σ(Жёлтые×0.5 + Зелёные×1.0)/с.
- **Энергия:** Ресурс обслуживания. Расход = Σ(активные узлы)/с. При дефиците узлы отключаются по приоритету. Стартовый прирост +100/с.
- **Старт:** 1000 Инфы, 100 Энергии. Возврат 50% при сносе.

### 5. ️ Рельеф и Климат
- Барьеры физически останавливают сигнал. Сеть строится в обход или через высотные позиции.
- Пустыня временно сужает радиус. Снег повышает шанс перехода в Красный режим.

### 6. 🎮 Интерфейс и Управление
- **HUD:** Баланс Инфы/Энергии, таймер, скорость (Пауза/1x/2x/5x).
- **Ввод:** Drag & Drop карты, зум колесиком. Клик по объекту → контекстная панель (режимы, снос, статистика). Миникарты нет.

### 7.  Метрики
- **Покрытие:** % активных пользователей в зоне сети.
- **Задержка:** Зависит от хопов и рельефа. Снижает доход при высоких значениях.
- **Стабильность:** Избыточность путей. Защищает от разрывов при событиях.

### 8. 🛠️ Режим игры
- Свободная песочница без миссий и дедлайнов.
- Автосохранение в LocalStorage, ручной экспорт/импорт конфигураций.

## 📐 Архитектура и Реализация (Modules)

Проект построен на модульной архитектуре. Каждый модуль изолирован, общается через главный оркестратор или события, и следует единым стандартам кодирования.

### 📂 Текущая структура (`/WeaveNet/src/modules/`)
```
/modules/
├── game.js          # 🟢 Главный оркестратор: цикл, ввод, ресурсы, связь модулей
├── hexGrid.js       # 🟢 Логика сетки: axial-координаты, соседи, валидация размещения
└── gridRenderer.js  # 🟢 Рендеринг: Canvas отрисовка гексов, объектов, курсоров, эффектов
```

###  Описание реализованных модулей

#### 1. `game.js` — Игровой оркестратор
- **Задача:** Инициализация систем, обработка ввода пользователя, управление ресурсами, запуск игрового цикла.
- **Ключевые методы:** `constructor()`, `setupEventListeners()`, `placeBuilding()`, `gameLoop()`, `updateResourceUI()`.
- **Состояние:** Хранит `resources`, `selectedBuildingType`, ссылки на `HexGrid` и `GridRenderer`.

#### 2. `hexGrid.js` — Логика гексагональной сетки
- **Задача:** Управление структурой карты, расчёт координат, поиск соседей, проверка валидности размещения.
- **Структура данных гекса:** `{ id, q, r, neighbors[6], building, x, y }`
- **Координаты:** Axial-система (flat-top). Конвертация в пиксели: `x = q * width`, `y = (r + q/2) * height`.
- **Состояние:** Генерирует начальную сетку, предоставляет методы `getHex()`, `getAllHexes()`, `isValidPlacement()`.

#### 3. `gridRenderer.js` — Графический рендерер
- **Задача:** Отрисовка игрового поля на HTML5 Canvas с частотой ~60 FPS.
- **Ключевые методы:** `render()`, `drawHex()`, `drawBuilding()`, `drawBuildCursor()`, `resizeCanvas()`.
- **Оптимизация:** Перерисовка только изменённых областей, кэширование путей отрисовки, адаптация под размер окна.

###  Поток данных и Игровой цикл
```
[User Input] → game.js (обработка логики) 
              ├→ hexGrid.js (проверка координат/валидации)
              → gridRenderer.js (визуальное обновление)
                      ↓
                [Canvas Display]
```
- Цикл работает через `requestAnimationFrame()`.
- Логика обновления состояния отделена от отрисовки.
- Состояние гексов синхронизируется между `HexGrid` и `GridRenderer` на каждом кадре.

## 📝 Технический Стандарт

### 🗣️ Формат комментариев в коде
Обязателен для всех файлов, классов и сложных блоков:
```js
// [ЧТО] Функция расчёта координат пикселей из axial-системы.
// [ЗАЧЕМ] Позволяет точно позиционировать гексы на canvas без смещений.
// [PLAN] Добавить поддержку смещения камеры для панорамирования.
function axialToPixel(q, r) { ... }
```

### 🐛 Обработка ошибок
- Критические баги фиксируются в `/workspace/ERROR/`.
- После исправления в файл добавляется запись: `✅ Исправлено: [описание]`.
- Файл `ERROR.txt` очищается или архивируется при полном закрытии известных проблем.

### 📜 Версионирование и Changelog
- Формат версии: `v0.0.0.0.0` (каскадный инкремент, сброс при 9).
- Файл `ChangeLog` создаётся в `/workspace/ChangeLog/` после каждого завершённого этапа.
- Содержит: дату, выполненные задачи, технические детали, статус сборки, затронутые файлы, историю последних 5 версий.

###  Правила работы с планом
- Задачи выполняются строго по возрастанию номеров из `plan.txt`.
- Перед написанием кода проверяется `MODULES_README.md` на соответствие архитектуре.
- После реализации обновляются: `plan.txt` (статус `[+]`), `TODO.md` (очистка), `ChangeLog` (новый файл).
