## 📦 МОДУЛЬ 1: 🗺️ Карта и Ландшафт

### 1.1. Гексагональная структура мира
- [ ] **1.1.1. Базовая сетка:**
    1. Мир состоит из шестиугольных клеток (гексов), каждая имеет уникальные координаты.
    2. Расстояния и радиусы измеряются в количестве гексов, а не в пикселях.
    3. Гекс — минимальная единица пространства: на одной клетке может быть только один объект.
    4. Сетка бесшовная: переход между гексами не создает разрывов в логике игры.
    5. Границы карты четко определены: за пределами — «пустота», недоступная для взаимодействия.
- [ ] **1.1.2. Форма и масштаб:**
    1. Карта визуально повторяет контуры территории России (узнаваемые очертания).
    2. Ключевые регионы выделены: Калининград, Крым, Байкал, Сахалин, Камчатка.
    3. Размер карты фиксирован: 200×200 гексов (достаточно для тактики, не перегружает восприятие).
    4. Масштабирование: игрок может приближать/отдалять карту, но логика гексов не меняется.
    5. Пропорции регионов сохраняются при зуме: пустыня остается пустыней, снег — снегом.
- [ ] **1.1.3. Координатная логика:**
    1. Каждый объект в игре привязан к центру гекса, дробные координаты исключены.
    2. Соседние гексы определяются по шести направлениям (север, северо-восток, юго-восток и т.д.).
    3. Расстояние между двумя гексами = количество шагов по сетке для перехода от одного к другому.
    4. Радиус действия любого эффекта (сигнал, покрытие) = круг из гексов на заданном расстоянии.
    5. Валидация: действие возможно только если целевой гекс существует и доступен.

### 1.2. Биомы и климатические зоны
- [ ] **1.2.1. Типы регионов:**
    1. Равнина (зеленая): базовый тип, без модификаторов к геймплею.
    2. Пустыня (песочная): визуальный стиль + косметические эффекты (пыль, марево).
    3. Снежные пустоши (белая): визуальный стиль + косметические эффекты (снег, иней).
    4. Биомы генерируются крупными кластерами, а не одиночными разрозненными гексами.
    5. Границы между биомами плавные: возможны смешанные зоны перехода.
- [ ] **1.2.2. Визуальная идентификация:**
    1. Каждый биом имеет уникальную цветовую палитру для мгновенного распознавания.
    2. Текстура гекса повторяет стиль биома (трава, песок, снег) без перегрузки деталями.
    3. При наведении на гекс отображается название его региона (тултип).
    4. Климатические эффекты (пыль, снегопад) — опциональны, не влияют на читаемость.
    5. Цветовая схема адаптируется под темную тему интерфейса игры.
- [ ] **1.2.3. Логика размещения:**
    1. Пользователи и постройки могут размещаться на любом типе региона, если гекс свободен.
    2. Преграды (горы, вода) могут появляться в любом регионе, но с разной вероятностью.
    3. Генерация мира: пропорции биомов настраиваются перед стартом (по умолчанию: 60% равнина, 20% пустыня, 20% снег).
    4. При перегенерации карты сохраняются настройки пропорций и размера.
    5. Биомы не меняются в процессе игры (статичный фон для тактики).

### 1.3. Преграды рельефа
- [ ] **1.3.1. Типы непроходимости:**
    1. Горы: гекс полностью блокирует прохождение сигнала и размещение объектов.
    2. Пропасти: гекс разрывает связь, сигнал не может пройти «сквозь пустоту».
    3. Водные преграды: гекс блокирует строительство, но сигнал может проходить над водой (если нет гор).
    4. Все преграды занимают ровно 1 гекс и визуально выделяются (текстура + высота/иконка).
    5. Преграды генерируются кластерами, формируя естественные барьеры и коридоры.
- [ ] **1.3.2. Влияние на геймплей:**
    1. Сигнал не проходит через горы и пропасти: сеть должна обходить эти гексы.
    2. Игрок вынужден планировать маршруты с учетом преград, что добавляет тактическую глубину.
    3. Преграды не потребляют ресурсы и не меняют статус в процессе игры.
    4. Визуально преграды «возвышаются» над картой, создавая ощущение объема.
    5. При наведении на преграду тултип сообщает: «Непроходимо / Нельзя строить».
- [ ] **1.3.3. Тактическое использование:**
    1. Горы могут служить естественным экраном: сигнал не уходит «за хребет».
    2. Пропасти создают узкие места: контроль над гексом у края пропасти дает стратегическое преимущество.
    3. Водные пространства можно «перепрыгивать» мощными вышками, создавая мосты связи.
    4. Игрок может использовать преграды для защиты сети: спрятать ключевой узел за горой.
    5. Баланс: преграды не должны делать карту непроходимой, всегда есть обходные пути.

## 📦 МОДУЛЬ 2: 👥 Население и Сетевые Связи

### 2.1. Пользователи как точки сети
- [ ] **2.1.1. Генерация и размещение:**
    1. На карте спавнятся точки-пользователи: в одном гексе может быть от 0 до 6 человек.
    2. Пользователи не появляются на гексах с преградами или уже размещенными постройками.
    3. Распределение неравномерное: кластеры пользователей в «городах», пустоши в «сельской местности».
    4. Каждый пользователь имеет уникальный внутренний ID, но для игрока это просто точка с числом.
    5. Начальное количество пользователей настраивается перед стартом (по умолчанию: 500–700).
- [ ] **2.1.2. Визуальное представление:**
    1. Группа пользователей в гексе отображается как одна точка с цифрой (количество).
    2. Размер точки масштабируется от 1 до 6: больше людей = крупнее иконка.
    3. Цвет точки соответствует статусу большинства пользователей в группе (приоритет: Зеленый > Желтый > Красный > Белый).
    4. При наведении тултип показывает детальную статистику: сколько Белых/Желтых/Зеленых/Красных.
    5. Анимация: точка мягко пульсирует, если в группе есть активные (Желтые/Зеленые) пользователи.
- [ ] **2.1.3. Привязка к сетке:**
    1. Каждый пользователь жестко привязан к гексу: не может «перетекать» между клетками.
    2. При пересчете логики игра проверяет статус гекса, а не каждого пользователя отдельно (оптимизация).
    3. Если гекс занят постройкой, пользователи в нем не генерируются и не учитываются.
    4. Пользователи могут «исчезать» (переход в Красный → Белый), но гекс остается на карте.
    5. Удаление пользователей возможно только через смену статуса, не через прямое удаление с карты.

### 2.2. Статусы пользователей и механика ретрансляции
- [ ] **2.2.1. Цветовая система статусов:**
    1. ⚪ Белый: нейтральный пользователь, не участвует в сети, не приносит доход.
    2. 🟡 Желтый: зараженное устройство, ретранслирует сигнал на 2 гекса вокруг, приносит +0.5 Инфы/сек.
    3. 🟢 Зеленый: доброволец, ретранслирует сигнал на 2 гекса, приносит +1.0 Инфы/сек, создает нагрузку.
    4. 🔴 Красный: заблокированное устройство, связь обрывается, доход не генерируется.
    5. Статус отображается цветом точки и, опционально, маленькой иконкой статуса рядом.
- [ ] **2.2.2. Механика ретрансляции:**
    1. Желтые и Зеленые пользователи передают сигнал на расстояние 2 гекса от своего гекса.
    2. Сигнал от пользователя может «подхватить» соседний пользователь или постройку в радиусе.
    3. Цепочка ретрансляции: Постройка → Пользователь → Пользователь → Постройка (и так далее).
    4. Сигнал не проходит через преграды (горы, пропасти) и не активирует Красных пользователей.
    5. Визуально цепочка отображается тонкой светящейся линией между центрами гексов.
- [ ] **2.2.3. Нагрузка и доход:**
    1. Зеленые пользователи создают «нагрузку»: их доход +1.0 Инфы/сек, но они требуют больше стабильности сети.
    2. Желтые пользователи приносят +0.5 Инфы/сек, но менее надежны (высокий риск поломки).
    3. Доход начисляется только если пользователь имеет активную связь с сетью (не изолирован).
    4. Нагрузка не расходует ресурсы напрямую, но влияет на метрику «Стабильность» (раздел 7).
    5. Баланс: Зеленые выгоднее, но требуют больше усилий по поддержанию связи.

### 2.3. Циклы смены статусов (Поведение пользователей)
- [ ] **2.3.1. Заражение (Белый → Желтый):**
    1. Белый пользователь автоматически становится Желтым, если его гекс попадает в радиус покрытия активной постройки.
    2. Заражение происходит мгновенно при размещении/включении постройки рядом.
    3. Если постройка выключена или снесена, Желтый пользователь не становится Белым сразу (инерция).
    4. Повторное заражение возможно, если пользователь стал Белым после поломки.
    5. Визуально: точка меняет цвет с серого на желтый с плавной анимацией.
- [ ] **2.3.2. Поломка (Желтый/Зеленый → Красный):**
    1. Желтый пользователь имеет ВЫСОКИЙ шанс (например, 5% за игровой час) перейти в Красный.
    2. Зеленый пользователь имеет НИЗКИЙ шанс (например, 0.5% за игровой час) перейти в Красный.
    3. Красный статус также может быть вызван внешними факторами (климат, события — раздел 5).
    4. При переходе в Красный связь через этого пользователя обрывается мгновенно.
    5. Визуально: точка темнеет, пульсация останавливается, линия связи исчезает.
- [ ] **2.3.3. Восстановление (Красный → Белый/Зеленый):**
    1. Желтый → Красный → Белый: после «починки» вирус исчезает, пользователь становится Нейтральным.
    2. Зеленый → Красный → Зеленый: после «починки» доброволец остается в сети (шанс 97–99%).
    3. Маленький шанс (1–3%) полного сброса Зеленого до Белого при длительной блокировке.
    4. Восстановление происходит автоматически через случайный промежуток времени (1–5 минут игрового времени).
    5. Игрок не может напрямую влиять на восстановление, только косвенно (улучшая стабильность сети).

## 📦 МОДУЛЬ 3: 💰 Экономика и Строения

### 3.1. Типы строений и их характеристики
- [ ] **3.1.1. Единый габарит:**
    1. Все постройки занимают ровно 1 гекс, независимо от типа.
    2. Визуальное различие достигается формой иконки и цветом, а не размером.
    3. Постройка не может быть размещена на гексе с преградой или другим объектом.
    4. При размещении постройки пользователи в этом гексе не удаляются, но перестают учитываться.
    5. Снос постройки мгновенно освобождает гекс, но не восстанавливает пользователей в нем.
- [ ] **3.1.2. Типы узлов:**
    1. Роутер (треугольник, зеленый): радиус 3 гекса, стоимость 30 Инфы, потребление 2 Энергии/сек.
    2. Вышка (пятиугольник, синий): радиус 6 гексов, стоимость 100 Инфы, потребление 8 Энергии/сек.
    3. Генератор (ромб, желтый): не излучает сигнал, производит +5 Энергии/сек, стоимость 60 Инфы.
    4. Все узлы имеют одинаковую высоту визуализации, различаются только формой иконки.
    5. Новые типы узлов могут быть добавлены в будущем без изменения базовой логики модуля.
- [ ] **3.1.3. Соединения и покрытие:**
    1. Постройки соединяются светящимися линиями с пользователями и другими постройками в радиусе действия.
    2. Линия рисуется, если между двумя объектами нет преграды и расстояние ≤ радиуса источника.
    3. Покрытие визуализируется как полупрозрачный круг с градиентным затуханием от центра.
    4. При пересечении кругов покрытия их яркость суммируется, показывая зоны усиленного сигнала.
    5. Линии и круги не перекрывают рельеф и иконки объектов, отображаясь на слое «под» ними.

### 3.2. Режимы работы построек
- [ ] **3.2.1. ⚪ Белый (Неактивен):**
    1. Постройка не потребляет Энергию, не излучает сигнал, не участвует в метриках.
    2. Используется для консервации: игрок может «заморозить» узел, чтобы сэкономить ресурсы.
    3. Пользователи в радиусе не заражаются, связи через этот узел не работают.
    4. Визуально: иконка серая, без свечения, радиус покрытия не отображается.
    5. Переключение в активный режим мгновенно, без задержек или дополнительных затрат.
- [ ] **3.2.2. 🟡 Желтый (Скрытый режим):**
    1. Радиус покрытия сокращен до 50% от базового (Роутер: 1.5 гекса, Вышка: 3 гекса).
    2. Потребление Энергии снижено на 50% (Роутер: 1/сек, Вышка: 4/сек).
    3. Заметность для «событий» (раздел 5) снижена, но не убрана полностью.
    4. Визуально: иконка желтая, радиус покрытия отображается пунктирной линией.
    5. Идеален для точечного расширения сети без перегрузки энергосистемы.
- [ ] **3.2.3. 🟢 Зеленый (Открытый режим):**
    1. Полный радиус покрытия (100%): Роутер 3 гекса, Вышка 6 гексов.
    2. Стандартное потребление Энергии (Роутер: 2/сек, Вышка: 8/сек).
    3. Максимальная эффективность: быстрее заражает пользователей, стабильнее связь.
    4. Визуально: иконка ярко-зеленая, радиус покрытия — сплошной градиентный круг.
    5. Рекомендуется для ключевых узлов, где важно максимальное покрытие.

### 3.3. Ресурсы и экономический баланс
- [ ] **3.3.1. Типы ресурсов:**
    1. Информация (Инфа): основная валюта, тратится на покупку и улучшение построек.
    2. Энергия: ресурс обслуживания, расходуется каждую секунду на активные постройки.
    3. Инфа не расходуется пассивно, только по действию игрока (постройка, снос, апгрейд).
    4. Энергия генерируется пассивно: +1/сек за каждые 10 активных пользователей в сети.
    5. Генераторы добавляют фиксированный бонус: +5 Энергии/сек за каждый активный генератор.
- [ ] **3.3.2. Доход и расход (формулы):**
    1. Доход Инфы = (Количество Желтых × 0.5) + (Количество Зеленых × 1.0) в секунду.
    2. Расход Энергии = Σ(потребление каждой активной постройки) в секунду.
    3. Баланс Энергии: если расход > дохода, Энергия уходит в минус, но не ниже 0.
    4. При Энергии = 0 постройки отключаются по приоритету: сначала Роутеры, потом Вышки, Генераторы — в последнюю очередь.
    5. Стартовый баланс: 1000 Инфы, 10 Энергии (хватает на 3–5 Роутеров или 1 Вышку).
- [ ] **3.3.3. Управление инфраструктурой:**
    1. Покупка: игрок выбирает тип постройки → кликает на свободный гекс → списывается Инфа → постройка появляется.
    2. Снос: игрок выбирает постройку → подтверждает снос → возвращается 50% стоимости в Инфу → гекс освобождается.
    3. Переключение режима: мгновенно, без затрат, доступно из контекстного меню постройки.
    4. Оптимизация: игрок может массово переключать режимы нескольких построек (выделение группой).
    5. Визуальная обратная связь: при нехватке Инфы кнопка постройки блокируется, при нехватке Энергии активные узлы мигают красным.
